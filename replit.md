# NEON AI DJ

Automates dancer rotations, manages music playback, and generates dynamic voice announcements for nightclubs.

## Run & Operate

- **Start API server**: `npm run api-server`
- **Start UI**: `npm run dev`
- **Build**: `npm run build`
- **Typecheck**: `npm run typecheck`
- **DB Push (for schema changes)**: `npm run db:push`
- **Required Env Vars**: `DB_PATH` (defaults to `./djbooth.db`, `/home/runner/data/djbooth.db` in prod)

## Architecture decisions

- **Persistent DJBooth Component**: The `DJBooth` component remains persistently mounted to preserve the audio engine state, ensuring seamless music transitions and announcement overlays.
- **Dual Audio Gain Control**: Music gain (`music_tracks.auto_gain`) and voice announcement gain (`voiceGainRef`) are managed independently, allowing for dynamic ducking and boosting of announcements without affecting music loudness.
- **Fail-Closed Song Preview**: The dancer phone/tablet song-preview UI (`DancerView`) is designed to only enable preview controls if the client is not on localhost AND the `isLocalhost` check from the API returns `false`, preventing booth kiosk misuse.
- **R2 Cloud Sync for Redundancy**: Music and voiceovers are synced to/from Cloudflare R2 on every boot and during service restarts, providing robust redundancy and ensuring content availability even if local files are lost.
- **Display Configuration by Port Name**: Display configuration uses `xrandr` based on port names (e.g., HDMI-1, HDMI-2) instead of orientation detection, providing more reliable multi-monitor setup on Dell units.

## Product

- Seamless music transitions with equal-power crossfading and audio ducking.
- Automated set management with 4-hour song cooldown, configurable songs-per-set, and interstitial break songs.
- Dynamic, human-sounding voice announcements generated by ElevenLabs TTS and OpenAI, adapting to club energy and operating hours.
- Centralized fleet management for multiple deployed units: remote monitoring, updates, content synchronization, and hardware health tracking.
- Autoplay Queue for music when no entertainers are present.
- Dancer data and PIN authentication.

## Gotchas

- **Dashboard "Errors: N" is a `console.error` counter, NOT an HTTP 404 count** (discovered May 17 on 002). The Fleet Command Center tile shows `Errors: N` in red. This value is `errorCounter` from `artifacts/api-server/server/index.js:125` — incremented every time `console.error` is called, sent in the heartbeat at line 2472. It is purely coincidental that tonight's value was 404. If this fires, **do NOT grep the journal for `404` or `not found`** — you'll get zero results and waste 10 minutes. Instead grep for `Express error` or the literal log lines that match `console.error` calls in the api-server code. Tonight's 404 was 404 instances of `Express error: Error: Cannot find module '../encodings'` from a missing `iconv-lite` package — see Open TODO about hardening the update script for full context.
- **Frozen mouse + touchscreen on Dell unit**: Caused by `gnome-shell` holding a stuck X input grab. Heavy-handed manual fix: `killall -HUP gnome-shell` (does not stop music). **Preventive auto-fix is in place fleet-wide as of May 14 (commit `5173a07`)**: `djbooth-touch-watchdog.service` cycles every touchscreen via `xinput --disable; sleep 0.5; xinput --enable` every 180s, which forcibly drops any stuck X grab on the touch device only (~500ms invisible blip, no GUI flicker, no music interruption). Vendor-agnostic — recognizes ILITEK, Goodix, Weida, eGalax, ELAN, FocalTech, Wacom, Atmel maXTouch, HID-multitouch, Siliconworks/SiW, plus any device with the legacy `Abs MT Position X` property. Logs to `/tmp/djbooth-touch-watchdog.log`. If touchscreens still freeze with this running, escalate by checking the log; tier-2 recovery would add `killall -HUP gnome-shell` and tier-3 a kiosk relaunch.
- **`xinput --enable` resets the Coordinate Transformation Matrix to identity** (silently un-mapping the touchscreen from the kiosk monitor). Discovered live on 003 May 14 — touch kept "breaking" because our own touch-watchdog was un-mapping the touch device every 180s. **Fix landed May 14 commit `5173a07`**: the watchdog now re-applies `xinput map-to-output <id> <KIOSK_OUTPUT>` immediately after every successful enable, so touch self-heals every 180s. Confirmation log line: `remap id=X (...) -> HDMI-2`. If this stops appearing in `/tmp/djbooth-touch-watchdog.log`, the watchdog has regressed.
- **Touchscreen registers in wrong place** (manual recovery if the watchdog isn't running): `DISPLAY=:0 xinput map-to-output <id> <monitor>` (per-unit `xinput list` and `xrandr` output needed).
- **The "Rotation Screen" button on the kiosk top bar is a working single-button recovery** (confirmed live on 002 May 23, 2026 ~05:00). Symptom that night: touch was being eaten by gnome-shell as 3-finger workspace-switch gestures (1-finger taps didn't reach Chromium either), mouse cursor frozen, crowd display rotation lost. Multiple terminal recoveries — `xinput --disable/--enable` + map-to-output, `bash ~/.djbooth-display-config.sh`, `killall -HUP gnome-shell` — had NO effect. DJ tapped the **Rotation Screen** button in the kiosk UI (top bar, next to Deactivate/Logout) and BOTH problems cleared simultaneously: crowd display flipped back to the correct rotation AND touch + cursor input started working again. Hypothesis: the button cycles the kiosk window's display assignment / re-grabs input, which forces gnome-shell to release whatever input lock it was holding. **For future touch-frozen incidents, tell the operator to push the Rotation Screen button on the kiosk BEFORE running any terminal recovery — it may fix everything in one tap.** If that doesn't work, fall back to the watchdog/HUP/unfreeze.sh chain. (Also: 002 was found that night to NOT have `djbooth-touch-watchdog.service` installed at all — the unit was missing from systemd. The watchdog and unfreeze.sh installation steps in `djbooth-update-github.sh` may have silently skipped on 002 during a past update; next time on 002 verify with `systemctl list-unit-files | grep djbooth` and reinstall manually if missing — install block is in `djbooth-update-github.sh` lines ~865-900.)
- **002's touch hardware is Weida Hi-Tech CoolTouchR (USB vendor 2575 product 0401), NOT Siliconworks/SiW like 003.** xinput on 002 lists it as two devices: a "...CoolTouchR System Mouse" entry (cursor-emulation interface) and a "...CoolTouchR System" entry (the real touch endpoint, used for map-to-output). On May 23, 2026 the real touch entry was id=20 but IDs drift across reboots — always re-detect. Critically: the Weida driver does NOT expose the `Abs MT Position X` xinput property the watchdog uses to auto-discover touch devices, so any auto-detector that scans for `Abs MT Position X` will MISS the Weida on 002 entirely (one such detector failed live on May 23). Use a wider scan: match by device name regex including "Weida" / "CoolTouchR", OR fall back to USB vendor 2575 lookup. The watchdog script in `artifacts/dj-booth/public/public/djbooth-touch-watchdog.sh` already includes Weida in its vendor-agnostic name regex per the May 14 fleet update, but verify on any unit where touch detection comes up empty.
- **OpenAI / R2 / apt failing with "fetch failed" / "Temporary failure resolving"**: Caused by Tailscale owning `/etc/resolv.conf` with `--accept-dns=true`. Fix: `sudo tailscale set --accept-dns=false` and reconfigure Wi-Fi connection with `ipv4.never-default yes ipv4.ignore-auto-dns yes`.
- **`killall -HUP gnome-shell` wipes xrandr state** — the standard input-grab fix ALSO drops `--primary`, drops `--rotate`, and repositions outputs. Safe when display config is default; **dangerous after any manual `xrandr --output ... --primary` or `--rotate` or `--pos`** because gnome-shell will reset them on HUP. Mitigation: re-apply xrandr right after HUP if you've customized display config, or use `xinput --disable <id>; sleep 0.5; xinput --enable <id>` on the touch device alone (less destructive than HUP, doesn't touch display state). Discovered May 11 during 003 monitor swap when HUP swapped the kiosk and rotation chromiums between monitors.
- **Hot-plugging a new monitor flips xrandr `primary` flag**: When a new HDMI monitor is added on Linux, the system may auto-promote a different connected output to `primary`. Since `chromium --kiosk` always lands on the primary output and ignores `--window-position`, the booth kiosk may end up on the wrong screen after a monitor swap. Fix: `xrandr --output <intended-kiosk-output> --primary` then relaunch chromium. Bake `--primary` into `~/.djbooth-display-config.sh` so it survives reboots.
- **Booth goes SILENT while the server stays healthy = Chromium suspended the Web Audio context, NOT a server/crash problem** (live incident May 28, 2026 on 002). Symptom: DJ reports music stopped, but `systemctl is-active djbooth` = `active` AND `curl -sf http://localhost:3001/__health` returns OK (echoes `OK`). The server was selecting tracks normally the whole time — the journal showed `🎵 selectTracksForSet` lines right up to and across the silence. Root cause: Chromium suspended its Web Audio `AudioContext`, which can be triggered by a GPU-process hiccup, memory pressure, or a USB audio device re-enumeration. That night the `C-Media Electronics Inc. USB Audio Device` re-enumerated during boot AND there was a stale `/tmp/chromium-rotation/SingletonLock: File exists (17)` in the journal — either can wedge audio. The booth UI keeps running and rotation keeps advancing; only the actual audio OUTPUT is dead. **The diagnostic that PROVES it's this and not a crash: the health curl returns OK while sound is dead.** **Do NOT restart the djbooth service for this** — the server is fine; restarting it does nothing for a suspended browser audio context (it'll come back `active`/OK and still be silent). **Manual recovery, fastest first:** (1) DJ taps the **Rotation Screen** button on the kiosk top bar; (2) reload the kiosk page from the unit shell: `DISPLAY=:0 xdotool key F5`; (3) full kiosk relaunch: `pkill -f chromium; sleep 3; nohup bash ~/djbooth-kiosk.sh > /tmp/kiosk.log 2>&1 &` then the DJ logs in + taps Start. **Auto-heal landed May 28, 2026 in `DJBooth.jsx`** — see the "Silent-booth self-heal" entry below; the in-app watchdog now reloads the kiosk automatically as a last resort when audio is wedged. - **Silent-booth self-heal (landed May 28, 2026 in `artifacts/dj-booth/src/pages/DJBooth.jsx` — NOT YET PUSHED, NOT ON ANY UNIT).** Extends the existing in-app audio watchdog (the `🐕 WATCHDOG` block) with a last-resort `window.location.reload()` to recover from a wedged audio output (the 002 May 28 suspended-Web-Audio-context failure above).
