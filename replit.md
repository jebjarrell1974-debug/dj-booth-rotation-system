# NEON AI DJ

Automates dancer rotations, manages music playback, and generates dynamic voice announcements for nightclubs.

## Run & Operate

- **Start API server**: `npm run api-server`
- **Start UI**: `npm run dev`
- **Build**: `npm run build`
- **Typecheck**: `npm run typecheck`
- **DB Push (for schema changes)**: `npm run db:push`
- **Required Env Vars**: `DB_PATH` (defaults to `./djbooth.db`, `/home/runner/data/djbooth.db` in prod)

## Architecture decisions

- **Persistent DJBooth Component**: The `DJBooth` component remains persistently mounted to preserve the audio engine state, ensuring seamless music transitions and announcement overlays.
- **Dual Audio Gain Control**: Music gain (`music_tracks.auto_gain`) and voice announcement gain (`voiceGainRef`) are managed independently, allowing for dynamic ducking and boosting of announcements without affecting music loudness.
- **Fail-Closed Song Preview**: The dancer phone/tablet song-preview UI (`DancerView`) is designed to only enable preview controls if the client is not on localhost AND the `isLocalhost` check from the API returns `false`, preventing booth kiosk misuse.
- **R2 Cloud Sync for Redundancy**: Music and voiceovers are synced to/from Cloudflare R2 on every boot and during service restarts, providing robust redundancy and ensuring content availability even if local files are lost.
- **Display Configuration by Port Name**: Display configuration uses `xrandr` based on port names (e.g., HDMI-1, HDMI-2) instead of orientation detection, providing more reliable multi-monitor setup on Dell units.

## Product

- Seamless music transitions with equal-power crossfading and audio ducking.
- Automated set management with 4-hour song cooldown, configurable songs-per-set, and interstitial break songs.
- Dynamic, human-sounding voice announcements generated by ElevenLabs TTS and OpenAI, adapting to club energy and operating hours.
- Centralized fleet management for multiple deployed units: remote monitoring, updates, content synchronization, and hardware health tracking.
- Autoplay Queue for music when no entertainers are present.
- Dancer data and PIN authentication.

## Gotchas

Quick symptom → fix below. **Full forensic detail (evidence, dates, root-cause analysis) for every item is preserved verbatim in `replit-archive.md` under "Gotchas — full forensic detail (moved from replit.md Jun 12, 2026)".**

- **Dashboard "Errors: N" is a `console.error` counter, NOT an HTTP 404 count** (002 May 17). Do NOT grep the journal for `404`/`not found` — grep for `Express error` or the literal `console.error` log lines.
- **Frozen mouse + touchscreen, music still playing = stuck X input grab.** Preventive `djbooth-touch-watchdog.service` cycles the touch device every 180s fleet-wide (since May 14 `5173a07`); heavy manual fix `killall -HUP gnome-shell` (doesn't stop music). Log: `/tmp/djbooth-touch-watchdog.log`.
- **`xinput --enable` resets the touch Coordinate Transformation Matrix to identity** (un-maps touch from the kiosk). Watchdog re-applies `xinput map-to-output <id> <KIOSK_OUTPUT>` after every enable; confirm line `remap id=X (...) -> HDMI-2`. If that line stops, the watchdog regressed.
- **Touch registers in wrong place (watchdog off):** `DISPLAY=:0 xinput map-to-output <id> <monitor>`.
- **"Rotation Screen" kiosk button is a one-tap recovery** for frozen touch + lost crowd-display rotation (002 May 23). Tell the operator to tap it BEFORE any terminal recovery. (That night 002 was missing `djbooth-touch-watchdog.service` — verify with `systemctl list-unit-files | grep djbooth`, reinstall from `djbooth-update-github.sh` if absent.)
- **002 touch hardware is Weida Hi-Tech CoolTouchR (USB 2575:0401), NOT SiW like 003.** It does NOT expose `Abs MT Position X`, so any detector scanning that property MISSES it — match by name "Weida"/"CoolTouchR" or USB vendor 2575. The real endpoint is the "...System" entry (not "...System Mouse"); IDs drift, re-detect each time.
- **OpenAI / R2 / apt "fetch failed" / "Temporary failure resolving" = Tailscale owns `/etc/resolv.conf`** (`--accept-dns=true`). Fix: `sudo tailscale set --accept-dns=false` + reconfigure Wi-Fi with `ipv4.never-default yes ipv4.ignore-auto-dns yes`.
- **`killall -HUP gnome-shell` wipes xrandr state** (drops `--primary`/`--rotate`/`--pos`). Dangerous after manual display config — re-apply xrandr right after HUP, or cycle just the touch device (`xinput --disable <id>; sleep 0.5; xinput --enable <id>`) which doesn't touch display state.
- **Hot-plugging a monitor flips the xrandr `primary` flag** → `chromium --kiosk` lands on the wrong screen. Fix: `xrandr --output <kiosk-output> --primary` then relaunch; bake `--primary` into `~/.djbooth-display-config.sh`.
- **Booth SILENT but server healthy (`systemctl is-active djbooth` = active AND `curl -sf http://localhost:3001/__health` = OK) = Chromium suspended its Web Audio context, NOT a crash.** Do NOT restart the service. Recovery fastest-first: (1) tap Rotation Screen button; (2) `DISPLAY=:0 xdotool key F5`; (3) full relaunch `pkill -f chromium; sleep 3; nohup bash ~/djbooth-kiosk.sh > /tmp/kiosk.log 2>&1 &` then DJ logs in + taps Start. In-app self-heal auto-reloads as last resort (DJ must re-login + tap Start).
- **`Failed to fetch` at login, service `active` but `curl http://localhost:3001/__health` empty/`BOOTH_DOWN` = node server crash-looping, usually a missing npm package on disk.** Confirm: `sudo journalctl -u djbooth -n 60 --no-pager`. Fix (no wipe): `cd ~/djbooth && NODE_ENV=development npm install --legacy-peer-deps && sudo systemctl restart djbooth && sleep 8 && curl -sf http://localhost:3001/__health && echo BOOTH_OK || echo STILL_DOWN`. Then check disk: `sudo dmesg | grep -iE 'ext4|i/o error|read error|sda|nvme' | tail -30`. (2 units have lost packages from disk: 002 `iconv-lite` May 17, 003 `express` May 28 — audit disk health fleet-wide.)
- **Silent-booth self-heal (in `DJBooth.jsx` — NOT YET PUSHED, NOT ON ANY UNIT):** in-app watchdog reloads the kiosk as a last resort on wedged audio (`ctx_suspended` ~12s, or `recovery_failed` 3 cycles), loop-guarded min 120s between reloads. ZERO changes to the LOCKED `AudioEngine.jsx`. After reload the DJ must re-login + tap Start. Test on 002 a full night before 003.
