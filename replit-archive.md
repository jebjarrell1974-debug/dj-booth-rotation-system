- **★ ROOT CAUSE FOUND (Jun 01, 2026) — the recurring "lost ~25,000 songs" on 003 was the UPDATE SCRIPT's rollback path, NOT power loss / drive failure / the 8AM reboot / filesystem corruption / R2 purge.** Evidence (003, from journal + live system state): at **May 31 08:31** a sync logged `25297 already local, 0 purged` — every song present. The machine **never rebooted after** (continuous uptime; the prior shutdown was clean — reached umount.target, synced filesystems). **Zero nvme I/O errors.** Music lives on the **single root partition** (`/dev/nvme0n1p2`, mounted `/`) — NOT a separate mount that could silently drop (confirmed with `findmnt` + `lsblk`: one nvme drive, one ext4 root, disk only 10% full). **Every** R2 sync logged `0 purged`, so the app deleted nothing (its purge also has a >20% safety brake). Yet by **Jun 01 02:36** all 25,297 were gone and re-downloading. Conclusion: the files were removed by a shell `rm -rf` off a RUNNING machine. **THE BUG:** `artifacts/dj-booth/public/public/djbooth-update-github.sh` backs up the app dir but DELIBERATELY skips `music`/`voiceovers`/`node_modules` (lines ~119-122 — far too large to copy). Its failed-update rollback did `rm -rf "$APP_DIR"; mv "$BACKUP_DIR" "$APP_DIR"` — which DELETED the live music/voiceovers/node_modules and restored a backup that never contained them → full 25k-song R2 re-download. It **recurs because updates HAVE failed to start before** — the May 17 `iconv-lite` and May 28 `express` missing-package crashes are exactly the kind of failure that TRIGGERS this rollback. This **supersedes the line-121 May-28 guess that FS corruption ate the music**: for the *music* loss the cause was this rollback; the separate *node_modules/express* loss may still be disk-related. **FIX (pushed to GitHub Jun 01, commit `61f94b9`, then hardened `768ec75`):** the rollback now moves `music voiceovers node_modules` from `$APP_DIR` into `$BACKUP_DIR` BEFORE the swap, **FAIL-CLOSED** — if any can't be moved to safety it ABORTS the rollback, reverts any partial moves, and leaves `$APP_DIR` fully intact rather than risk `rm -rf` on the only copy. **HOW IT REACHES UNITS (no manual step needed):** the script self-updates `~/djbooth-update.sh` from this exact GitHub source path at the START of every run (lines ~196-228) then re-execs — so the FIXED rollback runs from each unit's **very next update onward, automatically** (GitHub must be reachable on ethernet for the self-update; the bundle fallback also uses the source path). The OLD buggy rollback can never run again once GitHub has the fix, because the first pass `exit`s at the re-exec handoff (line ~229) before reaching its own rollback. **No dist rebuild needed** — the self-update pulls SOURCE; the stale `artifacts/dj-booth/dist/public/public/djbooth-update-github.sh` copy is never executed. **Deploy posture:** 003 was mid-R2-restore with auto-update DISABLED when this was found — do NOT run an update on 003 mid-restore; let the restore finish, then tap **Start Rotation**.
